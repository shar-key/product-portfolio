# Roadmap Artifacts

A roadmap is a prioritization argument, not a promise. Every roadmap 
I build is traceable to a scoring methodology, a validated research 
source, and a clear set of strategic bets. If a feature cannot be 
justified with evidence, it does not make the roadmap.

## How I Prioritize

I use two scoring frameworks depending on the context.

**RICE + Strategic Value**
Best for established product lines with measurable reach data.

Score = (Reach x Impact x Confidence x Strategic Value) divided by Effort

Reach: number of users affected per quarter
Impact: effect on the target metric (1 to 5 scale)
Confidence: how certain we are in the estimates (percentage)
Strategic Value: moat contribution, compliance requirement, or 
investor narrative weight (1 to 3 scale)
Effort: person-weeks to ship

**4D Founder Score**
Best for zero-to-one products where reach data does not yet exist.

Score = [(Investor Demo Value x 2) + (Revenue Unlock x 1.5) + 
(Moat Depth x 1.5) + (Blocking Other Capabilities x 2)] 
divided by Effort

Features scoring below 6.0 on either framework require a written 
justification before they enter the roadmap.

## Roadmap Format

I present roadmaps in table format only. No timeline swimlanes, no 
color-coded Gantt charts that imply false precision about future dates.

Columns: Issue ID, Title, Layer, Owner, Status, Milestone, Blockers

Layers define where a feature sits in the product architecture. Layer 0 
is infrastructure. Layer 1 is core user-facing features. Layer 2 is 
intelligence and analytics. Layer 3 and above is post-launch compounding 
features.

## What I Do Not Do

Build roadmaps driven by stakeholder pressure without scoring evidence. 
Commit to quarterly dates for features that have not been validated. 
Include features on the roadmap that do not have a corresponding PRD 
in progress. Present a roadmap without also presenting the prioritization 
rationale behind it.
